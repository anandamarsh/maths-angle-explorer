# Internationalisation (i18n)

The current codebase uses `src/texts.json` (a static JSON file) and `src/texts.ts`
(a simple `formatText()` helper). This spec describes the full i18n system to be
built, following the same pattern as `maths-game-template/specs/i18n.md` but adapted
for Angle Explorer's string inventory.

---

## Design principles

### The golden rule

Every user-visible string — UI labels, buttons, game feedback, question prompts, PDF
content, email body — must go through `t()`. A hardcoded English string in any of
these locations is a bug.

### Question prompt translation pattern

Question prompts look like natural language sentences. Different languages have
different grammatical structures and word orders — you cannot just swap words into an
English sentence structure.

**The solution: template keys with `{placeholder}` substitution.**

Each distinct question sentence structure is a separate translation key. The template
itself is translated into each language (including word order), and the variable values
are substituted at render time using `{placeholder}` tokens.

Angle Explorer's current level prompts are static strings with no variable
substitution, so they map directly to simple keys like `"level1.promptNormal"` and
`"level2.promptNormal"` — no placeholder tokens are needed for these.

**If future levels include variable angle values in the prompt** (e.g., "Aim at
{angle}°"), those must also use template keys with `{placeholder}` substitution. The
`t` function handles interpolation at call time. Do not build such strings with string
concatenation or template literals.

**How question generators must work:**

- All question generator functions accept `t: TFunction` as a parameter.
- They call `t("level1.promptNormal")` (or whichever key applies) to produce the
  rendered prompt string.
- The `prompt: string` field on the question object stores this pre-rendered,
  locale-specific string.
- This is "pre-baked" — the prompt is baked at generation time in the current locale,
  not re-translated on render.

**Functions that must accept `t: TFunction` (currently they do not — they read
`texts.levels["1"].prompts.normal` directly):**
- `makeL1Question(t: TFunction, ...)` in `src/game/angles.ts`
- `makeL2Question(t: TFunction, ...)` in `src/game/angles.ts`
- `makeQuestion(t: TFunction, ...)` in `src/game/angles.ts`

### PDF translation

`generateSessionPdf(summary, t)` accepts `t: TFunction` as a second parameter. Every
string drawn to the PDF — title, labels, objective text, encouragement messages,
footer — uses `t("pdf.someKey")`. No hardcoded English strings in the PDF generator.

### Email translation

`emailReport(summary, email, locale)` pre-translates all email strings using
`getT(locale)` before sending to the API. The `/api/send-report` serverless function
uses pre-translated strings when provided.

## Architecture

### Migration from `texts.json` (mandatory)

All strings currently in `texts.json` must be migrated to `src/i18n/en.ts`. The
`texts.ts` helper (`formatText`) is replaced by `TFunction` from `src/i18n/index.ts`.

This migration is mandatory. After migration, `src/texts.json` and `src/texts.ts`
must be removed. They must not coexist with the i18n system.

All call sites that currently use:
```ts
texts.levels["1"].prompts.normal
texts.rounds.monster.instruction
formatText(texts.generic.completeAllLevels.subtitle, { count: 2 })
```

Must be replaced with:
```ts
t("level1.promptNormal")
t("monster.instruction")
t("generic.completeAllLevelsSubtitle", { count: 2 })
```

### Files to create

```
src/
├── i18n/
│   ├── types.ts          # Translations type + TFunction type
│   ├── en.ts             # English strings (source of truth)
│   ├── zh.ts             # Chinese (Simplified)
│   ├── hi.ts             # Hindi
│   └── index.ts          # I18nProvider, useT, getT, helpers
└── components/
    └── LanguageSwitcher.tsx  # Globe icon + dropdown
```

```
api/
└── translate.ts          # Vercel serverless: on-demand OpenAI translation
```

### Provider setup

Wrap the app root in `I18nProvider` (in `App.tsx`):
```tsx
import { I18nProvider } from './i18n'
export default function App() {
  return (
    <I18nProvider>
      <RotatePrompt />
      <ArcadeAngleScreen />
    </I18nProvider>
  );
}
```

### In React components

```ts
import { useT } from "../i18n";
const t = useT();
t("game.fire")                         // → "Fire!"
t("game.correctAngle", { angle: 90 }) // → "Correct! 90°"
```

### Outside React (PDF, email)

```ts
import { getT } from "../i18n";
const t = getT(locale);
generateSessionPdf(summary, t);
```

### Locale persistence

Active locale stored as `localStorage["lang"]`, defaults to `"en"`.

## `TFunction` type

```ts
export type TFunction = (key: TranslationKey, vars?: Record<string, string | number>) => string;
```

Interpolates `{placeholder}` tokens. Unknown keys fall back to English.

## `LanguageSwitcher` component

Globe icon (🌐) button in the top-right toolbar cluster.

On click → dropdown with built-in languages + custom languages + "Other..." option.

**On-demand translation flow:**
1. User clicks "Other..." → dialog asks for language name
2. `POST /api/translate` with `{ targetLang: "French", strings: en }`
3. Response: `{ translations: {...}, langCode: "fr" }`
4. `cacheTranslation("fr", translations)` + `saveCustomLang("fr", "Français")`
5. `setLocale("fr")`

---

## On-demand translation API (`api/translate.ts`)

**Endpoint:** `POST /api/translate`

**Required env var:** `OPENAI_API_KEY`

**Request body:**
```ts
{ targetLang: string, strings: Record<string, string> }
```

**Response:**
```ts
{ translations: Record<string, string>, langCode: string }
```

**Model:** `gpt-4o-mini`, temperature 0.3, `response_format: { type: "json_object" }`

**System prompt rules:**
1. Preserve all `{placeholder}` tokens exactly
2. Do not translate URLs
3. Do not translate brand names: SeeMaths, Angle Explorer, DiscussIt
4. Do not translate angle type names if they are curriculum terms (e.g. ACUTE, OBTUSE, REFLEX)
5. Return `{ "translations": {...}, "langCode": "<ISO 639-1>" }`

Once added, `localApiPlugin()` in `vite.config.ts` must also mount a
`POST /api/translate` dev route (mirroring the Vercel function).

## PDF and email language support

`generateSessionPdf(summary, t, locale)` accepts `t: TFunction` and the active locale string. Every
string drawn to the PDF uses `t()` — no hardcoded English anywhere in the PDF
generator. For non-Latin scripts (Hindi, Chinese), a Unicode font is loaded from
`/public/fonts/` and embedded in the PDF so characters render correctly.

`emailReport(summary, email, locale)` pre-translates all email strings using
`getT(locale)` before sending to the API. The `/api/send-report` serverless function
uses pre-translated strings when provided, falls back to English otherwise.

---

## Adding new string keys

1. Add key to `src/i18n/types.ts` `Translations` type
2. Add English value to `src/i18n/en.ts`
3. Add translations to `zh.ts` and `hi.ts`
4. TypeScript will error if any locale file is missing the key
